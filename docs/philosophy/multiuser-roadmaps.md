# Мультиюзерность — named-roadmap

Над одним репозиторием работают несколько разработчиков, и у каждого — свой роадмап, свой запуск orchestrator, своя ветка. Разработчик декомпозирует задачи в собственный named-roadmap на своей машине, orchestrator исполняет их в его рабочей ветке, результат вливается в интеграционную ветку через PR. Роадмап-файлы разных разработчиков не пересекаются, поэтому не конфликтуют при мерже; общие поверхности проекта — [таблица Features и prune](#интеграционные-поверхности) — обслуживаются на интеграционной ветке.

## Раскладка

Named roadmap лежат в `.ai-factory/roadmaps/<имя>.md` — по файлу на разработчика. Дефолтная раскладка не меняется: `.ai-factory/ROADMAP.md` остаётся полноценным роадмапом, и проект с одним автором живёт на нём, не зная о мультиюзерности. Обе формы равноправны для всей семьи скиллов: скилл работает с тем файлом, который назван, а без аргумента — с дефолтным.

Тестовый роадмап — всегда напарник основного, в той же директории: у дефолтного это `ROADMAP_TESTS.md` рядом с `ROADMAP.md`, у именованного — `roadmaps/<имя>-tests.md` рядом с `roadmaps/<имя>.md`. Собственной деривации из git-идентичности у него нет — семья выводит его из роадмапа, который в работе (ключевые слова маршрутизации, слаг задачи), с той же owner-line и тем же single-writer.

Директория `roadmaps/` — точка перечисления: холодная сессия видит по её содержимому, чьи очереди работ живут в проекте, не раскапывая историю.

## Имя файла — из git-идентичности

Имя выводится автоматически: слагифицированный local-part из `git config user.email` — нижний регистр, каждая последовательность не-алфавитно-цифровых символов схлопывается в один дефис (`john.doe@example.com` → `john-doe.md`). Если email в конфиге отсутствует, fallback — слагифицированный `user.name`. Явный аргумент, как везде в семье, побеждает деривацию. Выведенный слаг, оканчивающийся на `-tests`, — жёсткий стоп с подсказкой передать имя явно: суффикс зарезервирован за тестовым напарником (`<имя>-tests.md`), и такой слаг сделал бы главный роадмап неотличимым от чужого тестового.

Ключ — именно email: это родной уникальный идентификатор авторства в git, он уникален по природе, стабилен, и его local-part пригоден для имени файла. `user.name` таким ключом не служит — свободная форма, пробелы и кириллица, нестабильность и реальные совпадения внутри команды.

## owner-line — коллизия ломается громко

Первая строка каждого named-roadmap объявляет владельца полным email'ом:

```
> Owner: john.doe@example.com
```

Строка пишется при создании файла. Каждый резолвер «моего» роадмапа — скилл или orchestrator — сверяет её с текущей git-идентичностью; несовпадение — жёсткий стоп с именем владельца и подсказкой: настроить git identity или передать имя роадмапа явно. Так остаточная коллизия выведенных имён (одинаковый local-part на разных доменах, совпадение fallback-имён) отказывает громко при первом же касании — а не проявляется тихой перекрёстной записью в чужую очередь.

Один разработчик с разными email'ами на двух машинах — две идентичности и два роадмапа; рабочая дисциплина — один email в git-конфиге везде.

## Инвариант single-writer

У каждого named-roadmap ровно один писатель — его владелец. Из этого инварианта следует вся бесконфликтность схемы: роадмап-файлы разных веток не пересекаются по содержимому, PR их не сталкивает.

Зависимости между роадмапами — дисциплина разработчиков, не механизм: задачи тесно связаны внутри одного роадмапа, связи между роадмапами допустимы как осознанные и координируются очерёдностью PR. Семья tradeoxy с четырьмя связанными роадмапами — рабочий образец этой дисциплины.

## Разрешение целевого файла в семье скиллов

Вся роадмап-семья параметризована целевым файлом, и named-roadmap входит в неё тем же аргументом. Порядок разрешения един:

1. явный аргумент — путь или имя файла — побеждает всегда;
2. запрошен «мой» роадмап — имя выводится из git-идентичности, файл ищется в `.ai-factory/roadmaps/`, owner-line сверяется;
3. без аргумента — `.ai-factory/ROADMAP.md`.

Orchestrator получает путь к роадмапу из своих настроек — он задаётся вместе с окружением при поднятии рабочей машины. Настройка несёт три явных состояния: пустая — дефолтный `ROADMAP.md`; ключевое слово `my` — та же деривация из git-идентичности с проверкой owner-line (файла нет — громкий фолбэк на дефолтный роадмап: одна настройка обслуживает и мигрировавшие, и немигрировавшие проекты); явный относительный путь — берётся как есть, без owner-проверки.

Нумерация phase глобальна внутри файла — у каждого роадмапа своя ось номеров. Той же осью делятся task-spec: task-spec named-roadmap живут в его персональной поддиректории `.ai-factory/specs/<имя>/` — нумерация `<NN>` считается по директории, поэтому общей оси номеров между разработчиками нет и коллидировать нечему. Механизм нумерации при этом не меняется: engine и так нумерует внутри директории назначения, персональная папка — просто другое значение destination. Для читателей разделение невидимо — task-spec всегда разрешается через `Spec:`-тег contract-line с точным путём, никогда через перечисление директории. Тем же правилом делятся и артефакты orchestrator: у named-roadmap `plans/`, `plan-reviews/`, `reviews/` и `test-runs/` получают поддиректорию по стему его файла (`plans/john-doe/…`) с нумерацией по директории; дефолтная пара живёт в плоских директориях, как раньше. Плоская раскладка для всех невозможна: артефакты коммитятся вместе с task и разносятся мержами по веткам, так что общие папки становились бы объединением очередей всех разработчиков — дубли номеров от одновременных запусков, мерж-конфликты одинаковых имён и усыновление чужого плана resume-детектором orchestrator. Task-spec дефолтного роадмапа пишутся в плоскую `.ai-factory/specs/` как раньше — существующие проекты не замечают схемы.

## Интеграционные поверхности

Prune и таблица `## Features` в ARCHITECTURE.md — операции интеграционной ветки, один актор, после мержей; персонального prune не существует. Фичи — это фичи проекта, а не разработчика: авторство читается из истории коммитов. Единственность актора сохраняет обе поверхности single-writer — строка drop-history и фичевые строки не попадают в конкурентную запись.

Леджер drop-history общий на репозиторий: снапшот-коммит содержит все файлы, поэтому `git show <снапшот>:<путь роадмапа>` восстанавливает любой из роадмапов на момент любого prune. Семантика леджера — [roadmap-prune](../../src/skills/roadmap-prune/SKILL.md).

## Ось времени

В [дереве контекста](context-tree.md) роадмап — карта времени проекта, и мультиюзерность даёт этой карте ветвление, симметричное git'у: named-roadmap — ветки времени, рабочие буферы разработчиков; таблица Features — ствол, единая уплотнённая история; prune на интеграционной ветке — мерж времени. Настоящее проекта — объединение seam `[x]`/`[ ]` всех живых роадмапов, и вход в него — перечисление `roadmaps/`.

## Где нормировано

Модель времени одного роадмапа и правило seam — [дерево контекста](context-tree.md) и глобальный CLAUDE.md § «Grounding claims»; формат two-tier артефактов и разрешение целевого файла — `roadmap-engine`; семантика леджера и интеграционный prune — `roadmap-prune`; многорепозиторийный слой семьи, с которым эта схема ортогональна (grove делит семью по репозиториям, мультиюзерность — по писателям внутри одного), — [grove контекста](context-grove.md).
