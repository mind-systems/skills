# Цикл скилов — от замысла до сверки

Это описание того, как пакетом пользуются: в каком порядке скилы сменяют друг друга над одним проектом и что каждый шаг производит. Устройство самих скилов — в `skill-composition-model.md` (как сцепляются) и `skill-pyramid.md` (форма графа); дерево, которое они возделывают, — в `context-tree.md`. Здесь — только порядок.

У цикла два такта: **планирование живёт в чате, исполнение — в orchestrator** (отдельный CLI-процесс со своей документацией). Чатовые скилы читают и пишут артефакты в `.ai-factory/` и документацию, но прикладной код не трогают; orchestrator исполняет готовый роадмап задача за задачей и ничего не планирует. Граница жёсткая — это разные процессы.

## Замысел

Цикл начинается с разговора: задача описывается, всплывают ограничения и развилки, рождается общее понимание. Скилы ещё не в деле.

## phase — `roadmap-outline`

Замысел превращается в 5–15 крупных phase — phase заголовков с прозовыми интро. Каждая phase отвечает на вопрос «какую способность получает система». Эти заголовки — рамка всего дальнейшего: под ними выстроятся задачи, их именами ходят rescue и планнеры.

## Техзадание — `aif-docs`

Сразу после phase, до всякого кода, пишется ТЗ. Всё под `docs/` — governing-spec: поведение, протоколы, потоки данных, настоящим временем, существует код за ними или ещё нет. ТЗ phase попадает в её заголовок строкой `Governing spec:` — и с этого момента оно обязательное чтение каждого, кто phase трогает: планнеров и ревьюера orchestrator, rescue при разборе, архитектора при проходе. Агент воспринимает ТЗ через CLAUDE.md и индекс документации — поэтому их обновление входит в этот же шаг.

## Атомизация — `roadmap-decompose`

phase наполняются задачами `N.M` — атомарными, с одной причиной отката, каждая two-tier: contract-line плюс task-spec. Где резать, решает гейт атомарности («доставима ли первая половина без второй — осмысленно?»). ТЗ в этом шаге дописывается параллельно: декомпозиция вскрывает детали, которых в нём не было.

## Проход по phase — `agent-architect`

Тяжёлая phase проходится парой архитектор↔редактор: архитектор рассуждает, ревьюит и выдаёт work-orders, персистентный редактор-сабагент применяет и отчитывается фактами, архитектор проверяет по файлам. Скилы внутри прохода передаются по ссылке — путём до SKILL.md с аргументами, никогда пересказом. Пользователь правит развилки и владеет коммитами.

## Спецификация до кода — `roadmap-decompose-skeleton`

Внутри архитекторского прохода тяжёлые задачи получают второй разрез — по оси «специфицируй тихо-падающую поверхность прежде, чем её писать»: скелет (интерфейсы без тел), tests-first против поверхности скелета (фильтр `test-philosophy` — только тихо-падающее), contract-task для конкурентных зон. Сдержанность первого класса: простые задачи проход не трогает.

У этого шага есть обратное ребро, и оно — половина его ценности: скелетный разрез почти всегда вскрывает недописанное в ТЗ. Найденные дыры возвращаются в governing-spec правками — это самый ранний и дешёвый детектор недоспецификации, те же дыры у orchestrator на ревью стоят кратно дороже.

## Пины — `command-pin-gaps`

Последний проход перед исполнением: всё, где исполнителю пришлось бы угадывать, закрывается сейчас — value-дыры пинятся точным значением из кода с цитатой `file:line`, meaning-дыры (грани, развилки, неназванные инварианты, границы скоупа) — недостающей клаузой task-spec, выведенной из наблюдённого поведения; настоящие продуктовые решения уходят в явные блокеры, не в выдумку.

## Исполнение — orchestrator

Роадмап готов — orchestrator обрабатывает task по одной: план → ревью плана → имплементация → код-ревью → коммит, помечая каждую `[x]`. Агенты общаются файлами-артефактами в `.ai-factory/`; исполняется ровно записанное, без додумывания.

## Когда task не сходится — `task-rescue`, `task-rescue-audit`

Остановка на лимите итераций оставляет артефакты на диске. `task-rescue` диагностирует глубину корня (task-spec? план? код?), чинит на эту глубину и откатывает sidecar с артефактами в отремонтированное состояние. `task-rescue-audit` — взгляд снаружи на любую зациклившуюся или аномально долгую task, даже прошедшую: отличает сходимость-через-понимание от сходимости-через-измор; здоровая сходимость — ожидание, заплатка — исключение, которое стоит назвать. Обычный порядок: сперва rescue, следом audit по тёплым артефактам. Аудит при этом не привязан к циклу: он вызывается и сам по себе, в любой сессии, когда от кода пахнет костылями, которыми orchestrator обходил кривую архитектуру. Он только оценивает и рассказывает — ни во что не пишет.

## Тесты — `roadmap-test-coverage`

По накопленному стеку `[x]` задач выявляются области, которые стоит покрыть, — правилом `test-philosophy`: тестируется падающее тихо, пропускается падающее громко. Тест-задачи ложатся в отдельный `ROADMAP_TESTS.md` и исполняются тест-режимом orchestrator, где финальная проверка — реальный прогон тестов.

## Уборка и сверка — `roadmap-prune` → `aif-docs`

`roadmap-prune` сворачивает выполненные `[x]` задачи в именованные фичи с коммит-хешами в `ARCHITECTURE.md` и удаляет их из роадмапа вместе с отработавшими артефактами и их task-spec — карта снова короткая. Уходят все `[x]` без исключений; строку оставляет только явное указание пользователя, и она сохраняет свою task-spec.

У prune есть гейт: незапиненные deferred-наблюдения из ревью блокируют его целиком. Заблокированный prune паркуется — это штатное состояние ожидания, а не сбой: пользователь вызывает `command-handoff`, наблюдения с контекстом уезжают в выделенную сессию разрешения, та разбирает каждое — чинит, маршрутизирует в task-spec открытой задачи или отклоняет, — ставя пины по грамматике `orchestrator-artifacts`. prune перезапускается, когда гейт проходит честно; проход гейта — доказательство разрешения, никогда не его имитация.

Финальным идёт второй проход `aif-docs` — теперь как сверка: ТЗ, писанное до кода, сличается с тем, что реально построено. Контракт до и сверка после — два прохода одного скилла, разделённые исполнением.

## Схема

```
чат:    замысел
          │  roadmap-outline             → phase (заголовки + интро)
          │  aif-docs                    → ТЗ на phase (= Governing spec: phase)
          │  roadmap-decompose           → атомарные N.M задачи (∥ ТЗ дописывается)
          │  agent-architect             → проход по phase парой архитектор↔редактор
          │    └ roadmap-decompose-skeleton → skeleton / TDD / concurrency
          │         └──── дыры в ТЗ ────→ правки ТЗ (обратное ребро)
          │  command-pin-gaps            → закрыть места угадывания
          ▼
оркестр:  implement → план → ревью плана → имплементация → код-ревью → коммит
          │   task-rescue                ← task не сходится (чинит по глубине)
          │   task-rescue-audit          ← взгляд снаружи на зациклившуюся task
          ▼
чат:    roadmap-test-coverage → ROADMAP_TESTS.md → оркестр: test
          │  roadmap-prune               → фичи в ARCHITECTURE.md, чистка карты
          │    └── гейт заблокирован ──→ command-handoff → сессия разрешения
          │         └──── пины стоят ──→ повторный roadmap-prune
          │  aif-docs                    → сверка ТЗ с построенным
          ▼
        готово — следующий замысел входит сверху
```

Цикл повторяется поверх уже свёрнутой в `ARCHITECTURE.md` истории.
