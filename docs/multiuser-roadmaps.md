# Мультиюзерность — именованные роадмапы

Над одним репозиторием работают несколько разработчиков, и у каждого — свой роадмап, свой запуск оркестратора, своя ветка. Разработчик декомпозирует задачи в собственный именованный роадмап на своей машине, оркестратор исполняет их в его рабочей ветке, результат вливается в интеграционную ветку через PR. Роадмап-файлы разных разработчиков не пересекаются, поэтому не конфликтуют при мерже; общие поверхности проекта — [таблица Features и прун](#интеграционные-поверхности) — обслуживаются на интеграционной ветке.

## Раскладка

Именованные роадмапы лежат в `.ai-factory/roadmaps/<имя>.md` — по файлу на разработчика. Дефолтная раскладка не меняется: `.ai-factory/ROADMAP.md` остаётся полноценным роадмапом, и проект с одним автором живёт на нём, не зная о мультиюзерности. Обе формы равноправны для всей семьи скиллов: скилл работает с тем файлом, который назван, а без аргумента — с дефолтным.

Тестовый роадмап — всегда напарник основного, в той же директории: у дефолтного это `ROADMAP_TESTS.md` рядом с `ROADMAP.md`, у именованного — `roadmaps/<имя>-tests.md` рядом с `roadmaps/<имя>.md`. Собственной деривации из git-идентичности у него нет — семья выводит его из роадмапа, который в работе (ключевые слова маршрутизации, слаг задачи), с той же owner-строкой и тем же single-writer.

Директория `roadmaps/` — точка перечисления: холодная сессия видит по её содержимому, чьи очереди работ живут в проекте, не раскапывая историю.

## Имя файла — из git-идентичности

Имя выводится автоматически: слагифицированный local-part из `git config user.email` — нижний регистр, каждая последовательность не-алфавитно-цифровых символов схлопывается в один дефис (`kg.wmservice@gmail.com` → `kg-wmservice.md`). Если email в конфиге отсутствует, fallback — слагифицированный `user.name`. Явный аргумент, как везде в семье, побеждает деривацию. Выведенный слаг, оканчивающийся на `-tests`, — жёсткий стоп с подсказкой передать имя явно: суффикс зарезервирован за тестовым напарником (`<имя>-tests.md`), и такой слаг сделал бы главный роадмап неотличимым от чужого тестового.

Ключ — именно email: это родной уникальный идентификатор авторства в git, он уникален по природе, стабилен, и его local-part пригоден для имени файла. `user.name` таким ключом не служит — свободная форма, пробелы и кириллица, нестабильность и реальные совпадения внутри команды.

## Owner-строка — коллизия ломается громко

Первая строка каждого именованного роадмапа объявляет владельца полным email'ом:

```
> Owner: kg.wmservice@gmail.com
```

Строка пишется при создании файла. Каждый резолвер «моего» роадмапа — скилл или оркестратор — сверяет её с текущей git-идентичностью; несовпадение — жёсткий стоп с именем владельца и подсказкой: настроить git identity или передать имя роадмапа явно. Так остаточная коллизия выведенных имён (одинаковый local-part на разных доменах, совпадение fallback-имён) отказывает громко при первом же касании — а не проявляется тихой перекрёстной записью в чужую очередь.

Один разработчик с разными email'ами на двух машинах — две идентичности и два роадмапа; рабочая дисциплина — один email в git-конфиге везде.

## Инвариант single-writer

У каждого именованного роадмапа ровно один писатель — его владелец. Из этого инварианта следует вся бесконфликтность схемы: роадмап-файлы разных веток не пересекаются по содержимому, PR их не сталкивает.

Зависимости между роадмапами — дисциплина разработчиков, не механизм: задачи тесно связаны внутри одного роадмапа, связи между роадмапами допустимы как осознанные и координируются очерёдностью PR. Семья tradeoxy с четырьмя связанными роадмапами — рабочий образец этой дисциплины.

## Разрешение целевого файла в семье скиллов

Вся роадмап-семья параметризована целевым файлом, и именованный роадмап входит в неё тем же аргументом. Порядок разрешения един:

1. явный аргумент — путь или имя файла — побеждает всегда;
2. запрошен «мой» роадмап — имя выводится из git-идентичности, файл ищется в `.ai-factory/roadmaps/`, owner-строка сверяется;
3. без аргумента — `.ai-factory/ROADMAP.md`.

Оркестратор получает путь к роадмапу из своих настроек — он задаётся вместе с окружением при поднятии рабочей машины. Настройка несёт три явных состояния: пустая — дефолтный `ROADMAP.md`; ключевое слово `my` — та же деривация из git-идентичности с проверкой owner-строки (файла нет — громкий фолбэк на дефолтный роадмап: одна настройка обслуживает и мигрировавшие, и немигрировавшие проекты); явный относительный путь — берётся как есть, без owner-проверки.

Нумерация фаз глобальна внутри файла — у каждого роадмапа своя ось номеров. Той же осью делятся спеки: спеки именованного роадмапа живут в его персональной поддиректории `.ai-factory/specs/<имя>/` — нумерация `<NN>` считается по директории, поэтому общей оси номеров между разработчиками нет и коллидировать нечему. Механизм нумерации при этом не меняется: движок и так нумерует внутри директории назначения, персональная папка — просто другое значение destination. Для читателей разделение невидимо — спека всегда разрешается через `Spec:`-тег контракт-строки с точным путём, никогда через перечисление директории. Тем же правилом делятся и артефакты оркестратора: у именованного роадмапа `plans/`, `plan-reviews/`, `reviews/` и `test-runs/` получают поддиректорию по стему его файла (`plans/kg-wmservice/…`) с нумерацией по директории; дефолтная пара живёт в плоских директориях, как раньше. Плоская раскладка для всех невозможна: артефакты коммитятся вместе с милстоуном и разносятся мержами по веткам, так что общие папки становились бы объединением очередей всех разработчиков — дубли номеров от одновременных запусков, мерж-конфликты одинаковых имён и усыновление чужого плана resume-детектором оркестратора. Спеки дефолтного роадмапа пишутся в плоскую `.ai-factory/specs/` как раньше — существующие проекты не замечают схемы.

## Интеграционные поверхности

Прун и таблица `## Features` в ARCHITECTURE.md — операции интеграционной ветки, один актор, после мержей; персонального пруна не существует. Фичи — это фичи проекта, а не разработчика: авторство читается из истории коммитов. Единственность актора сохраняет обе поверхности single-writer — строка drop-history и фичевые строки не попадают в конкурентную запись.

Леджер drop-history общий на репозиторий: снапшот-коммит содержит все файлы, поэтому `git show <снапшот>:<путь роадмапа>` восстанавливает любой из роадмапов на момент любого пруна. Семантика леджера — [roadmap-prune](../src/skills/roadmap-prune/SKILL.md).

## Ось времени

В [дереве контекста](context-tree.md) роадмап — карта времени проекта, и мультиюзерность даёт этой карте ветвление, симметричное git'у: именованные роадмапы — ветки времени, рабочие буферы разработчиков; таблица Features — ствол, единая уплотнённая история; прун на интеграционной ветке — мерж времени. Настоящее проекта — объединение швов `[x]`/`[ ]` всех живых роадмапов, и вход в него — перечисление `roadmaps/`.

## Где нормировано

Модель времени одного роадмапа и правило шва — [дерево контекста](context-tree.md) и глобальный CLAUDE.md § «Grounding claims»; формат двухуровневых артефактов и разрешение целевого файла — `roadmap-engine`; семантика леджера и интеграционный прун — `roadmap-prune`; многорепозиторийный слой семьи, с которым эта схема ортогональна (роща делит семью по репозиториям, мультиюзерность — по писателям внутри одного), — [роща контекста](context-grove.md).
