# Skills Roadmap

> Generic AI Factory skills — reusable slash-command packages for Claude Code.

## Pyramid rewrite — the skill package on the written philosophy

The philosophy is written — [Skill pyramid](../docs/skill-pyramid.md), [Context tree](../docs/context-tree.md), grounded in the [Skill composition model](../docs/skill-composition-model.md) — but every skill and command in `src/` predates it. This direction rewrites the whole package to the pyramid. Hard rules for the direction: rewrites are behavior-first — a rewritten skill is unverified until a live run compares its output to the pre-rewrite baseline; mechanism never inlines into a top and a top never absorbs an engine's mass; `loads:` declarations and the engines' reverse-graph markers remain the dependency map throughout; the repo docs stay the philosophy's single home — rewrites are audited against them and never restate them; a link into a skill body only where it is load-bearing.

### Phase 1 — Rewrite the skill package to the pyramid

Opens with its own substrate (1.1–1.3) — the philosophy substrate lands before the rewrites read it; each rewrite then passes its live baseline before the next starts. The problem today: the package grew before the philosophy was named, and the pyramid inverts in places — `command-handoff` (140 lines) outweighs the `note` engine (115) it delegates to; top-level skills still carry mechanism of their own; the newest layers (folder-carried style, the walkable tree) are encoded only in the newest files. The contracts this phase rests on: a top-level skill is a **lens** — route plus policy, tens of lines, with ~80 as the aspiration and never a clamp; the mass migrates down into the engines (`note`, `roadmap-engine`); every artifact writer funnels through `note` via its three hooks — destination, template, verbosity — with no file mechanics of its own; each rewritten skill passes its live baseline comparison before the next rewrite starts.

## Multiuser — named roadmaps

Several developers work one repository: each holds an own named roadmap, runs the orchestrator on it in an own branch, and merges results into the integration branch via PR. Governing spec: [Мультиюзерность — именованные роадмапы](../docs/multiuser-roadmaps.md) — the ratified design: `.ai-factory/roadmaps/<slug>.md` with the slug derived from git `user.email`, the `> Owner:` first line as the loud collision stop, the single-writer invariant, target-file resolution argument → mine → default, per-roadmap spec subdirectories `.ai-factory/specs/<name>/`, integration-branch prune/Features over a repo-wide ledger. Hard rules for the direction: defaults are byte-stable — a project without `roadmaps/` sees zero behavior change (lazy migration); the orchestrator's roadmap path is an environment setting in its own repository — out of this direction's scope.

## Plan-layer citation boundary
