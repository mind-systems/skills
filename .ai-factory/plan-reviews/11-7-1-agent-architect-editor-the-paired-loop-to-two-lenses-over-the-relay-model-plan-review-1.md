**1. (Medium) The spec's headline acceptance gate — the live paired-loop dry-run — is dropped from the plan's Verification.**
The spec makes the live run the *primary* correctness check, twice: Guard "Pyramid-baseline — unverified until a live paired-loop run compares behavior against the pre-rewrite baseline," and Verification item "Live paired-loop dry-run: the editor's analysis prompt equals the relayed user message … reasons independently on analysis and applies faithfully on apply … self-verify + flag-every-judgment-call retained." The plan's Verification keeps only `wc -l`, the two grep gates, and the vocabulary check. Those confirm *text was removed and renamed* — they cannot confirm the *behavior change* (relay-on-analysis, two editor modes) actually holds or that no load-bearing discipline was lost in compression, which is exactly what the spec says greps alone do not settle. Since this milestone introduces a behavior change (not pure compression), passing on greps + wc alone would mark the task done on the very evidence the spec calls insufficient. A live architect↔editor session is genuinely not automatable by an orchestrator implementer, so the fix is not "run it in-task" but "carry it forward honestly": add the dry-run to the plan's Verification as an explicit **manual post-merge gate** (with the spec's success criteria), rather than silently omitting the spec's caveat so the reader knows the rewrite is unverified until that run.

**2. (Low) The editor's "The work-order is the unit" discipline is neither in the retain list nor the cut list — and its "always exactly one fenced block" framing collides with analysis mode.**
Task 2 enumerates retained disciplines meticulously (self-verify, flag judgment calls, escalate/fix-and-say, skill-by-reference, spawn-once, commit-on-permission, English) and an explicit cut list (two essays, Counterpart, ground-truth/commit restatements). The current editor's `## The work-order is the unit` section — "don't merge two work-orders, don't split one," "always exactly one fenced block" — appears in neither list, so the implementer must guess whether and how to preserve it. It is load-bearing for *apply* mode, but "always exactly one fenced block" is literally false for *analysis* mode, where the relayed unit is the user's own message (the spec: "the editor's analysis prompt equals the relayed user message"), not a fenced work-order. The plan should name this section explicitly and instruct that the don't-merge/don't-split discipline be recast so it applies to the round's unit in both modes without asserting "fenced block" of a relayed analysis message. As written the omission risks either a dropped apply discipline or a retained clause that contradicts the new analysis mode.
