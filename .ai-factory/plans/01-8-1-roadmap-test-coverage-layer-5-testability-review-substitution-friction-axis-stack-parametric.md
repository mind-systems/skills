# Plan: roadmap-test-coverage: Layer 5 testability review — substitution-friction axis, stack-parametric

## Context

Layer 5 of `roadmap-test-coverage` judges testability on a binary DI-presence axis written entirely in TS/NestJS idiom, which misfires on other stacks. This task rewrites that single agent prompt template so the axis is a graded friction-to-substitute-in-a-test judgment made in the stack's own test idiom, while the verdict token grammar Layers 6/8 consume stays byte-identical.

**The discriminator this task turns on.** One criterion governs the whole axis, ratified in the spec's gradient table: **does the API offer a parameter for the thing a test must vary?** Code that receives its varying inputs as arguments and touches module-level names only as static reference data is `clean` — whatever those names are. Varying data or a collaborator the API offers no parameter for — acquired at construction or reached inside the call — is the middle band, however cheap the patch: a `monkeypatch` is still friction, because the test author must know the acquisition site and patch it instead of passing a value. "Cannot be stood up without patching" is deliberately *not* the criterion — defensive acquisition (an `.exists()` guard returning a default) keeps construction from failing without offering any parameter, and it must still flag. Construct names (module singleton, `os.environ`, `datetime.now()`) are stack-conditional illustrations, never the band definition.

  **Bind `<stack>` to `$STACK` — the parameter needs a source.** Unlike `<area name>` and `<file path>`, which the orchestrator lines at `:160-161` already imply ("one agent per area … reads the source file"), `<stack>` has no anchor.   **Placeholder form — pinned, do not improvise.** No `$VAR` appears inside any agent prompt fence in this file; Layers 4, 5 and 7 use angle-bracket placeholders exclusively, and the one `$TEST_CMD` inside a fence is angle-wrapped as `<$TEST_CMD>` (`:211`).   The new prompt must:
  - Open with the header `Testability review for: <area name>` / `Stack: <stack>` / `Source file: <file path>`.
  - State the axis explicitly: **how much friction it costs to substitute each dependency in a test**, judged in the given stack's own test idiom (pytest `monkeypatch` / `unittest.mock.patch`, jest module mocks, etc.) — not whether the stack's DI mechanism was used.
  - Teach the three bands **by one criterion — does the API offer a parameter for the thing a test must vary**:
    - **received** — the varying inputs arrive as constructor params, function arguments, or injected collaborators; a test drives the code by passing values, and any module-level names it reads are static lookup data → no friction → `clean`;
    - **no parameter for what varies** — the code acquires varying data or a collaborator at construction or inside the call, and the API offers no parameter to supply it instead; the only way to vary it is to patch the acquisition site, however cheap that patch → friction → `needs-refactor`, framed as friction;
    - **unoverridable in this stack** — no patch point exists at all → blocking → `needs-refactor`, strong.
  - **Distinguish a collaborator from a helper — the criterion is incomplete without this clause.** A reached function whose behavior is fully determined by the parameters the caller passes it is **not** a substitution point: the test already controls it by controlling those parameters, so calling it directly costs nothing. A reached function counts as a collaborator only when it supplies varying data or an effect the test must control *and* the API offers no parameter for it. Without this, "acquires … a collaborator … inside the call" reads literally onto every helper call and every `open()`, and code that is genuinely `clean` flags — see Task 3's case 3, which reaches `_load_csv` and must still come back `clean` because the path it passes is the caller's own parameter.
  - State that the *same* construct sits in different bands in different stacks and call sites, so the names below are illustrations of the criterion, **never a universal checklist** — and that the stack's patchability sets the `<sentence>` register (friction to drop, never "untestable"), not the verdict. In particular: reading a module singleton, `os.environ`, `get_config()`, or `datetime.now()` is by itself **not** the middle band; it lands there when it is data the behavior varies on and the API offers no parameter to supply it instead. Whether the acquisition can *fail* is irrelevant — a defensive `.exists()` guard returning a default adds no parameter and does not clear the flag.
  - Keep the teardown check (fire-and-forget async, timers without a returned handle) and the unsafe-cast check (`(req as any).user`, `(metadata as any)[KEY]`) as **stack-conditional** illustrations, explicitly scoped to TS/Node stacks. Same for `process.env` / `Date.now()` — TS/Node examples only, never universal lines.
  - Pin the `<sentence>` register for the middle band: name the friction and the fix ("self-acquires its config at construction; inject it to drop test-setup friction"). Forbid "not injected" / "not DI", and forbid "untestable" for anything overridable in the stack.
  - Carry the mandatory scope fence as its own explicit sentence: flag coupling **only to the degree it taxes test setup**. What the fence clears is code with **nothing to substitute** — varying inputs arrive as parameters and module-level names are read only as static reference data; that is `clean`. **Keep this as a distinct proposition from the design-smell exclusion** — coupling that is merely ugly and costs test setup nothing is a separate, second case that is out of scope and belongs to a code-review lens, not a test-coverage planner; do not merge the two into one sentence where "that" ends up referring back to the parameter-receiving (`clean`) case instead of the design-smell case. **It never clears a patchable acquisition of varying data — cheapness of the patch is not the fence** (spec `:52`: "the patch is easy" is not the fence). Do not phrase this as "a self-acquired dependency that is near-zero-friction to substitute is `clean`" — that sentence contradicts the parameter criterion's own "however cheap the patch" clause and re-admits the rejected "patchable → clean" reframe (spec `:15`).
  - **Stay off the silent-failure axis** (spec `:61` — two orthogonal axes, two owners). The prompt must not ask the agent to weigh failure-loudness, silent failure, or whether the area deserves a test at all. That discriminator is `test-philosophy`'s and is applied in Layer 3; Layer 5 judges only the cost of authoring the test.
  - Use registry vocabulary in all new prose (`docs/using-the-language.md` § "What conforms" binds skill bodies) — in particular no `milestone` and no `spec note`. Roadmap task 17.3 sweeps this same file for exactly those synonyms; prose written here in retired words becomes debt 17.3 must re-clean.
  - End with the return-format block **byte-identical** to today:
    ```
    Return exactly one line in this format:
      clean | <area name>
      OR
      needs-refactor | <area name> | <one sentence: what to refactor and why>
    ```
