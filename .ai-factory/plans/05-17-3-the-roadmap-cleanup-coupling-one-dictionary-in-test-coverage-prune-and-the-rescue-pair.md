# Plan: 17.3 — The "roadmap cleanup" coupling: one dictionary in test-coverage, prune, and the rescue pair

## Context

Retire the synonyms `milestone`→`task` and `spec note`→`task spec` across the four maintenance-flow skills (`roadmap-test-coverage`, `roadmap-prune`, `task-rescue`, `task-rescue-audit`), normalizing odd prose casing. Naming-only conformance to [reserved-words](../../docs/reserved-words.md).

**Behavior: no losses, one additive gain.** The spec's guard is "zero behavior change", and every edit here holds to it with one recorded exception — §3's dual form at `roadmap-prune:325` additionally teaches prune to recognize a `## Tasks` heading it would today miss. Nothing any skill recognizes, scans for, or emits today stops working; the carve-outs in §1/§3/§4 exist precisely to guarantee that. Stating this plainly rather than repeating "zero behavior change" verbatim, because the guard's purpose is *no silent regression*, and an unstated addition is the kind of drift the guard is written against.

Each carve-out is a place where the mechanical `milestone`→`task` rewrite would be wrong, or needs a different justification; the reasoning below is the verification of the spec's instruction, not an override.

1. **`roadmap-prune:75` — `out of scope for this milestone` is a scan literal, not prose. DO NOT EDIT.** The spec's prune paragraph freezes this line on the same grounds; what follows is the verification. Line 75 sits inside a list of *pre-standardization marker phrases* (`latent`, `forward risk`, `no action needed`, `out of scope for this milestone`, `flagging so Phase`, `Surface this to the orchestrator`) that Step 4.6 greps for inside **historical review files written before the deferred-observations standardization**. Those files literally contain the word "milestone"; rewriting the needle to "task" silently stops matching them. `roadmap-prune` is a generic skill run against any project's `.ai-factory/`, so the corpus is external to this repo — its absence from this repo's swept artifacts is not evidence the phrase is dead. This is the exact failure the task's own guard forbids ("a rename that changes behavior is a bug"). Same class as `## Deferred observations`: a scanned literal, immune to the vocabulary contract.

2. **`task-rescue:356–357` — one cross-repo citation, one ground-truth sync covering both of its halves.** The sentence mirrors the orchestrator's sidecar-step table: "This table mirrors `_validate_sidecar_step()` / `_detect_milestone_step()` in `orchestrator/main.py`". Both halves are stale against the same ground truth, verified directly in the orchestrator repo: **both functions live in `orchestrator/resume.py`** — `_validate_sidecar_step` at :11 and `_detect_task_step` at :169, the milestone-named function having already been renamed upstream. So the edit is:

   - identifier `_detect_milestone_step()` → `_detect_task_step()`
   - path `orchestrator/main.py` → `orchestrator/resume.py`

   **Both on one rule, not two.** This is a fixed-form ground-truth sync, not a vocabulary edit — the identifier changes because the orchestrator's code says so, and the path changes for exactly the same reason. An earlier revision of this plan corrected the identifier and deferred the path as "outside the naming-only boundary"; that was incoherent, since the identifier edit had already stepped outside that boundary deliberately. Worse, it was not neutral: a freshly verified identifier sitting beside a knowingly wrong path raises the sentence's apparent authority while leaving the misdirection intact, giving a reader *more* reason to trust the bad path. The fix is inside the task's file boundary, on a line the task already edits — a finding, not a deferral. Record once in the commit rationale as a cross-repo citation sync.

3. **`roadmap-prune:325` — `## Milestones` is an external recognition literal; it gains the registry name but keeps the legacy one.** The spec and the contract line both mandate the dual form (`## Tasks` primary, legacy literal retained); this section is its grounding. Read in place, line 325 is not this skill's own heading — it describes the shape of a *target project's* ROADMAP.md, listing two alternatives the agent matches against an external file: `## <Direction name>` (a placeholder) and `## Milestones` (a concrete literal). `roadmap-engine` defines no `## Milestones` heading anywhere — its flat fallback is specified as "flat unnumbered bullet format" with no heading name (`roadmap-engine:120`) — so this literal exists precisely to recognize legacy or hand-written roadmaps carrying it. A bare rename means prune runs against such a project, fails to recognize the section, and deletes nothing: a silent no-op prune. That is the same external-corpus argument as §1, and it applies here identically.

   **Resolution — dual form, not a carve-out:** rewrite the phrase to name the registry word *and* retain the legacy literal, e.g. ``a flat `## Tasks` (or legacy `## Milestones`) list``. This satisfies the direction (the registry name is now the primary form the reader and the agent see), preserves the legacy scan target, and additionally makes prune recognize a `## Tasks` heading — which today it would not, a latent gap once the vocabulary lands project-side. Behavior is additive: strictly more roadmaps recognized, none lost.

4. **`task-rescue:10` — the quoted trigger phrase `"milestone failed"` is user input. DO NOT EDIT.** The line reads `trigger phrases: "rescue", "milestone failed", "pipeline stopped"` — these are quoted *user utterances*, listed so the always-loaded description matches what a user actually types. [using-the-language](../../docs/using-the-language.md) puts this on the explicitly non-conforming side: "what the system reads — user input is mapped by context, never conformed. A user types 'milestone', and the agent understands." Users will keep typing "milestone failed": the word survives in every historical roadmap, review file, and habit. Rewriting the literal degrades invocation matching on the exact phrase the skill was authored to catch, silently. Same external-corpus argument as §1 and §3, in its purest form — this corpus lives in the user's head, where no sweep reaches.

   **Resolution — leave it verbatim, not a dual form.** Two grounds, both checked: (a) **landed precedent.** Task 17.2 faced this exact class and the contract line records the ruling — "the user-trigger word 'milestones' in outline's `Use when` list **stays** (contract exempts input)". Verified in the file: `roadmap-outline:3` still reads `Use when user says "roadmap", "project plan", "milestones"` after 17.2 landed. The direction has already decided this class, and consistency with the sibling task governs. (b) **Phase 14.2 pulls the other way.** It targets these four descriptions for altitude levelling and names the rescue pair the "exhaustive pole" likely to come *down* toward the reference; padding the trigger list with an alternate form now adds words a planned phase will want to remove.
   **In contrast, the surrounding description prose does conform** — `~8` "checks downstream milestones" is ordinary prose about the concept, not a quoted utterance, and takes `tasks`. The boundary is the quotation marks.

**Byte-frozen everywhere:** `## Deferred observations` (prune 35/42/71, audit 58), `- Affects: …`, `PLAN_REVIEW_PASS` / `REVIEW_PASS`, the `` Spec: `` and `Governing spec:` header tags, the `.ai-factory/specs/` path, every `loads:` line, `name:`, `allowed-tools`.
