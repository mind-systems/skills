# Plan: 15.1 — agent-architect: relay by `::` marker, not by classification

## Context
Replace the architect's per-message classification judgment ("is this editor-bound work or conversation to me?") with a mechanical trailing-`::` marker, leaving exactly two editor channels: the `::` relay and the authored apply work-order. Body-only edit to `src/skills/agent-architect/SKILL.md`.

- Logging: minimal

## Assumptions
- `agent-architect/SKILL.md` frontmatter has **no** `loads:` field today (verified: the only `loads:` occurrence is body prose about the *editor's* skill-by-reference resolution). The spec's "`loads:` byte-identical" therefore means "still absent" — do not add one.
- Sections are identified by their headings and opening words, not by line numbers: the spec's own ranges are off by one against the file, and each edit shifts what follows. Locate every target by its quoted opening words.

## Decisions taken here (not left to the implementer)
- **The relay section's heading changes** to `## Relay on the marker; author a prompt in exactly one case`. "Relay by default" is the exact proposition Phase 15 deletes — after this change the default (no trailing `::`) is *never forward*, and a heading asserting the opposite of the section beneath it is the precise failure mode this task exists to close. Spec 70 and this plan both refer to the section by its old heading; that is expected drift in already-written artifacts, not a defect to reconcile.
- **The editor is spawned by the first channel-message — either channel.** The first `::` relay or, where no relay has arrived by greenlight time, the first authored apply work-order; the message's content *is* the spawn prompt, so nothing is ever authored *for* a spawn. The ordinary invocation (`/agent-architect phase 8`, no marker) spawns nothing: the architect works alone on the unit `$ARGUMENTS` names until a channel-message exists, and says so. This is spec 70's lifecycle rule; the work-order limb closes the session where no relay ever arrives (otherwise the greenlit work-order has no editor to go to and the architect's own hands are forbidden — a deadlock).
- **Death is loud; no auto-replay; the digest stays as an unsent recovery note.** Per spec 70: on a stale handle at post-compact recovery or a failed send, the architect reports to the user **before anything is sent onward** — an undelivered payload is never replayed into a fresh spawn, because the user phrased it for a warm context. The user re-phrases a relay as a self-contained spawn prompt; an apply work-order may be resent as-is (self-contained by contract). The respawn is the next channel-message after the report, never eager with authored prose. The digest in the pre-compact handoff is the architect's own recovery note and is never sent to the editor.
- **The warm-editor rule is retired in the spec, not just in the plan.** Spec 70 asserted "a warm editor is never re-sent a skill it already read" in three places (§Edits, §Guards, §Verification); all three are **amended in this task** to the initiative-scoped form — the architect never adds a skill reference the payload does not contain, and a slash-command the user typed expands unconditionally, warm editor or not. Reason: the warm phrasing requires tracking what the editor has already read and gating a send on it — per-message judgment, the exact surface Phase 15 deletes — and it collides with the unconditional expansion rule. Handoff 22's violation #2 was expansion with *no slash-command in the message*, so keying expansion to a command's presence closes it mechanically. A task spec still asserting a rule the implementation deliberately omits is a defect to reconcile, not stale description — hence the amendment rather than a deviation note.
- **Skill-by-reference expansion never applies to the `/agent-architect` invocation itself.** It applies to a skill the payload names *as the work* (spec 70's example, `/roadmap-decompose-skeleton phase 8 ::`). Expanding the invoking command would tell the editor to read `agent-architect/SKILL.md` — spawning a second architect.
