## Context

Conform the `aif` skill family to the harness wiring contract at the source: the rules artifact lands at canonical `.ai-factory/RULES.md`, and the `.ai-factory/config.yaml` apparatus — which the orchestrator never reads — is retired outright, with `aif-architecture`/`aif-docs` falling back to their already-written default paths.

- Docs: no

**The same false-green trap runs along three further axes, and Task 9 covers all four.** A residue naming the retired config need not contain the string "config" at all:
- **`paths.docs` / `paths.architecture`** — config keys that mention neither "config" nor "config.yaml" (`aif-docs` L41 is the live case).
- **"resolved architecture path"** — `aif-architecture` phrasings whose only resolution source was the config (L124, L126, L136, L141, L152).
- **`.ai-factory/rules/`** — a residue naming only the retired *directory*, which a `rules/base\.md` pattern does not match (`aif/SKILL.md` L116).

When sweeping, the **grep result is the authority** and the enumerated line lists in Tasks 5 and 6 are aids to it, never a ceiling over it: each list was built from a grep aimed at *that* axis, but a phrasing the grep returns is in scope even if no list happens to name it. Settle the full grep result; the only occurrence a grep surfaces that stays byte-identical is the one named exclusion — L73, the `$ARGUMENTS` pattern sense.

**The inverse trap, which is the more dangerous one.** Every axis above is a stale string that must *disappear*. This task also carries a rule that must *survive*, and deletion-shaped work is exactly where a survival requirement goes quietly: the **fixed-English invariant for generated artifacts** currently lives at `aif/SKILL.md` L47 and `aif-architecture/SKILL.md` L25 — both inside ranges this task deletes, and L25 is the only statement of English anywhere in its file. No zero-hits check can notice an invariant that vanished, because its absence *is* the clean state every other check is looking for. Tasks 3 and 5 restate it; Task 9 asserts it as a deliberately **non-zero** grep; Task 10 exercises it live. Read any instruction below to delete a language line as an instruction to first settle where the invariant it carried is going to live.

Also confirmed live: `grep -rln` across `src/` shows `config-template.yaml` / `config-persistence.md` / `update-config.mjs` are referenced only by `aif/SKILL.md` and each other — retiring them strands nothing, exactly as the spec states. (Plan review verified this more broadly across the grove and the `orchestrator/` repo, with the same result.)

**One deliberate consequence to accept, not repair:** `config-persistence.md` also carries the git-workflow-detection dialog (`git.enabled` / `git.base_branch` / `git.create_branches`). Those values were only ever written *into* `config.yaml` and read back by nothing, so they die with it. Do not relocate that dialog into `SKILL.md` — retiring it is in scope and re-homing it is not.

**Register guard (applies to every task below):** these are targeted edits to living instruction text, not rewrites. Match each file's existing imperative/declarative voice and leave untouched sentences byte-identical.

### Ruling: the `<resolved docs dir>` phrasings in `aif-docs` survive

`aif-docs/SKILL.md` says "the resolved docs directory" or `<resolved docs dir>` at roughly twenty places (L19, 22, 24, 41, 49, 81, 100, 121, 143, 145, 154, 156, 162, 170, 238–241, 245, 259, 271). **These stay, deliberately** — do not sweep them to a literal `docs/`.

The reason: "resolved" still has a live referent after this task. `aif-docs` writes to `docs/` by default *or* to the project's existing docs directory when one is already in use — Step 0 scans for exactly that. What the retirement removes is one *source* of the resolution (`paths.docs`), not the resolution itself. Rewriting twenty lines to a hard-coded literal would both overreach the register guard and assert something less true than what is there now.

What Task 6 does instead is make Step 0 the **definition site**: it states once what the docs directory resolves to now, so every downstream "resolved docs directory" reads against a definition that exists. Only the phrasings that name the retired mechanism by name — `paths.docs`, `config` — are edited.
