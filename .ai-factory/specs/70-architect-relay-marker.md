# agent-architect: the `::` marker replaces relay-classification judgment

Phase 15 of the paired-loop direction. One skill body, one change of kind: the architect stops *classifying* which user messages go to the editor and starts *reading a marker*. Grounded in [handoff 22](../handoffs/22-architect-violated-relay-discipline-contaminated-editor.md) — a live architect self-report of the exact failure this closes.

## Why prose keeps failing

The skill already states the relay discipline clearly, and the architect still violated it repeatedly (authored briefs, leaked its inventory and a collision-hint, planned to re-send a warm skill). The handoff's own remedy — add a named anti-pattern warning — pulls the **same lever that already failed**: more prose describing a negative constraint against a strong default ("de-risk the editor by pre-loading context — control masquerading as helpfulness"). The architect understands the rule (it filed the report) but cannot hold it *in the moment*.

The structural cause: the current skill makes the architect **decide**, per message, "is this an editor-bound relay or conversation aimed at me?" (SKILL.md lines 65–74, "Relay is scoped to editor-bound work… is your own standing judgment"). That judgment is the drift surface. Remove the judgment and there is nothing to drift.

## The design — two channels, one marker

The architect has **exactly two** ways to send anything to the editor:

1. **The `::` relay.** The *only* path that carries the user's own words. When a user message **ends with `::`**, the entire message before the trailing marker is the relay payload. The architect: strips the marker, translates the payload to English if needed, expands a slash-command in the payload to skill-by-reference (the one existing transformation), and sends it — adding **nothing**: no findings, no inventory, no collision-hint, no checklist, no framing, no "produce X and Y." A message that does **not** end with `::` is conversation to the architect and is **never** forwarded.
2. **The authored apply work-order.** Unchanged from today: the architect authors it itself, after the user's explicit greenlight, pinning every value/path/exact-string and the guardrails. This is the one place the architect writes its own prose to the editor.

The marker is **unconditional** — a trailing `::` always means relay; there is no "did the user really intend it" check, because that check *is* the classification we are deleting. A user who needs a message to literally end in `::` phrases around it. Zero judgment is the whole point.

The marker **trails, never leads**, for a mechanical reason: a relay payload often opens with a slash-command (`/roadmap-decompose-skeleton phase 8 ::`), and the harness invokes a skill only when the slash-command opens the message — in the *architect's own* session, where the architect needs it too (it reasons over the same target in parallel with the editor). A leading `::` would displace the command and strand the architect without the invoked skill; trailing, the message head stays exactly what the harness expects. One consequence: a message is one channel, whole — there is no mixed form where part of a message is conversation and part is relay. `::` anywhere but the very end is literal payload text; a user who wants to talk to the architect *and* relay something sends two messages.

The editor is spawned by the **first channel-message, whichever channel it is** — the first `::` relay or, where no relay has arrived, the first authored apply work-order — and messaged thereafter; the message's content *is* the spawn prompt, so nothing is ever authored *for* a spawn. The ordinary `/agent-architect` invocation carries no marker and spawns nothing: the architect works alone until a channel-message exists, and says so. **There is no third channel** — no authored spawn brief, no digest re-brief.

**Death is loud.** On discovering the editor gone — a stale handle at post-compact recovery, or a failed send — the architect reports to the user **before anything is sent onward**; a failed or undelivered payload is never auto-replayed into a fresh spawn, because the user phrased it for a warm context. The user, now knowing the editor will be fresh, re-phrases a relay as a self-contained spawn prompt; an apply work-order may be resent as-is (self-contained by contract). The respawn is the next channel-message after that report — never an eager respawn with authored prose. A respawned editor resumes through the same two channels, because `editor.md` already contracts every round self-contained (*"a fresh editor picking up from the next round alone must succeed just the same"*). This makes the authored channel exactly one thing (the apply work-order), and its hard rule the one that defeats the handoff-22 leak: the authored channel carries pinned edit-instructions only, **never the architect's analysis of an analysis-target** — target-analysis reaches the editor solely as the user's own `::`-relayed words.

## Guards

- **Body only, no `description:` edit.** Keeps clear of Phase 14.1 (which levels the altitude of `agent-architect`/`editor` descriptions and re-verifies against the final field) and of Phase 10.1 (which treats both files as audit-clean for vocabulary). This change is behavioral and orthogonal to both; it may land in any order relative to them, but a later reader of those phases should know the body moved.
- **The marker is unconditional and trailing-only.** No guessing whether the user "meant" the trailing `::`; `::` anywhere but the message end is literal text. Reintroducing intent-detection reintroduces the drift this deletes.
- **Slash-command in a payload** still expands to skill-by-reference; `loads:` engines still resolve on the editor's side (not pre-resolved). The trailing position exists for this case: the command stays at message head, so the harness invokes the skill in the architect's own session.
- **Two channels, no third.** The editor is spawned by the first channel-message of either channel; death is loud — reported to the user before anything is sent onward, with no auto-replay of an undelivered payload; a respawned editor resumes through the two channels, never an authored brief. The architect never adds a skill reference the payload does not contain (the second handoff-22 violation) — that message was the bare "go, phase 7" with no slash-command in it, so keyed to a command's presence in the payload the failure is structurally impossible. Not phrased as a warm-editor rule: that form requires tracking what the editor already read, reintroducing per-message judgment.
- **Mechanics settled; wording rides the pipeline.** The `::` rule (a trailing `::` triggers a relay, the whole message before it is the payload, no trailing `::` → no relay) is fixed. The exact English of the rewritten section is the implementer's, checked by the orchestrator's normal review stage — **no ratification gate**: a stop for out-of-band approval only strands an autonomous run.
- **No behavior verified by run** — a skill body change is verified by reading it back against this spec and against handoff 22's "correct behavior" list.
