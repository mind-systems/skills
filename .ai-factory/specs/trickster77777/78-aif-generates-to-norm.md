# aif family: stop generating retired artifacts, align to the canonical harness contract

Phase 19. Governing spec: `docs/sakshi-harness/sakshi-harness.md` — its wiring contract states the correct, present-tense shape of a project's coordination layer (CLAUDE.md opens on content, AGENTS.md a symlink, `.ai-factory/RULES.md` the orchestrator's canonical channel, no mention of config.yaml/rules/base.md/DESCRIPTION.md because they are absent from the correct state). Today's five Phase-2 passes (mind/tradeoxy/repo-stats-herald/observability/digital_ocean) found the same two defects landing in every project family, always from one source: `aif` and its `references/` templates still generate the rules file at the orchestrator-invisible `.ai-factory/rules/base.md`, and a `.ai-factory/config.yaml` the orchestrator never reads. Every future `/aif` run regenerates exactly what Phase 2 spent a day removing. This task conforms the generators to the harness contract at the source, so no future project needs this cleanup.

The "CLAUDE.md Generation" section (L31–39) already opens the generated file on a contentful project title + first `## section` and needs no opening-prohibition — a live `/aif` run on an empty scratch dir confirmed clean output; the `# CLAUDE.md`/"This file provides guidance…" line comes from a pre-existing `/init` file, not from this section. Its only stale content is the L33 ordering clause naming `.ai-factory/config.yaml` and `.ai-factory/rules/base.md`. "Language Resolution" (L43–67) persists `language.ui`/`language.artifacts` to `.ai-factory/config.yaml` and resolves them back from it every run. Each of the three modes' step lists (Mode 1 L71–120, Mode 2 L124–150, Mode 3 L154–189) carries an explicit "Persist config.yaml" step and a "Resolve Language Settings" step; Mode 1 Step 7 additionally writes `.ai-factory/config.update.json`, invokes `update-config.mjs`, and deletes the temp payload (items 3–5), then "Create rules/base.md" (item 6). Numerous "[Localized … in `language.ui`]" placeholders throughout assume a resolved, persisted language value.

## Change

- **CLAUDE.md-generation ordering clause** (`aif/SKILL.md` L33): drop the `.ai-factory/config.yaml` item and rename the `.ai-factory/rules/base.md` item to `.ai-factory/RULES.md`, keeping the "first artifact in every mode" rule intact. No opening-prohibition is added — `aif` invoked fresh already opens `CLAUDE.md` on a contentful title + first `## section` (confirmed live); the boilerplate line is a `/init` artifact, not something this section emits.
- **Rename the rules-artifact target** everywhere in `aif/SKILL.md` and `aif/references/rules-generation.md`: `.ai-factory/rules/base.md` → canonical `.ai-factory/RULES.md` (the `description:` field, the numbered overview, the "Create rules/base.md" step — renamed step, `.ai-factory/` created directly, no `rules/` subdirectory — the template scaffold's own header `# Project Base Rules` is unaffected, only the file's path/name changes).
- **Remove config.yaml generation entirely**: delete every "Persist config.yaml" step across all three modes, Mode 1 Step 7's config.update.json/update-config.mjs/cleanup sub-items, and the `references/config-template.yaml`, `references/config-persistence.md`, `references/update-config.mjs` files themselves (retire outright — nothing else references them).
- **Remove the language-pin apparatus**: delete the "Language Resolution" section's persisted-config machinery (`language.ui`/`language.artifacts` write-and-resolve-from-`config.yaml`, the "Resolve Language Settings" step in each mode, the AskUserQuestion language-selection prompt). `aif` communicates in the ambient chat/repository language going forward — no persisted state, no resolution step. The **fixed-English-headings rule for generated artifacts is unchanged** — only its persisted-config mechanism is removed, not the invariant. Because the generating skills are **independently invocable and do not share a body, the invariant is restated in each skill that writes a fixed-English artifact, not in one place**: `aif` carries it (for `CLAUDE.md`/`RULES.md`/`AGENTS.md`) inside `## CLAUDE.md Generation`, its natural home now that `## Language Resolution` is gone; `aif-architecture` carries it (for `ARCHITECTURE.md`) in its own Step 0, since `/aif-architecture` runs standalone and never loads `aif`'s body. "Ambient language" applies only to user-facing chat and confirmations — **never to the text of a generated fixed-English artifact**. The `## Language Resolution` heading itself is **dropped**, not left standing over content that no longer resolves anything. Every "[Localized … in `language.ui`]" placeholder in user-facing confirmation text becomes plain ambient-language prose (no bracketed placeholder, no `language.ui` reference).
- **`aif-architecture/SKILL.md` Step 0** and **`aif-docs/SKILL.md` Step 0**: delete the "read `.ai-factory/config.yaml` FIRST" step and its resolved-values framing; the existing "if config.yaml doesn't exist, use defaults" path (already written in both) becomes the only path, unconditionally — paths at their stated defaults. Language resolves per the per-generator rule above, and the two files land **opposite**: `aif-architecture` keeps its generated `ARCHITECTURE.md` in **fixed English** — its Step 0 restates the invariant, replacing the retired `en` config key that used to pin it — while user-facing prompts go ambient; `aif-docs` **splits**, because it writes into fixed-English artifacts as well as into docs. Its **docs content goes ambient** — generated docs are absent from the fixed-English artifact list, and the file's own rule ("if the project README is in Russian, write docs in Russian") already prescribes ambient, so dropping the `en` default resolves a pre-existing contradiction inside the file toward that rule. But the **`## Documentation` index sections `aif-docs` writes into `CLAUDE.md` and `AGENTS.md` stay fixed English** — heading and table text — and its Step 0 states this alongside the ambient rule. Those two files are on the fixed-English artifact list, `aif-docs` claims those sections in its own ownership block and may create `CLAUDE.md` outright, and `aif`'s `## CLAUDE.md Generation` already names `## Documentation` (with the `| Doc | What it covers |` header) as a fixed English heading. A blanket "fully ambient" here would have the two skills prescribing opposite languages for the same bytes — `aif` English, `aif-docs` ambient — in a project whose README is not English.

Any line enumeration is an aid to that grep, never a ceiling over it — a phrasing the grep surfaces is in scope even if no list names it, and verification asserts the sole survivor by its content, never by a line number (the edits shift every line below them).
- **Preserve register** — match the existing imperative/declarative voice of each file; this is a targeted edit, not a rewrite.
